Project Meeting 1

Date: 11-Mar-2026 (Wednesday)

Time: 4:00pm – 5:00pm

Attendees: Ricardo Moreno, Li Gen, Huang Bingqian

1.1 Topic Selection and Initial Dataset Exploration

  • Ricardo shared that he had explored several datasets that could potentially be used for the group project and introduced the COFINFAD (Colombian Fintech Financial Analytics Dataset) provided by the professor. Ricardo explained that the dataset contains information on customer demographics, financial attributes, product usage, and behavioural indicators, which can support multiple customer analytics tasks.

  • Li Gen reviewed the dataset structure and noted that the dataset contains two components: a customer dataset and a transaction dataset, with the customer dataset containing around 54 variables suitable for analysis. But the transaction dataset was very large, which may increase the difficulty of analysis and implementation. And also pointed out that the transaction dataset only covers one year of data, which limits its usefulness for long-term trend analysis.

  • Bingqian asked whether the transaction dataset might still be useful for analysing transaction behaviour over time. After discussion, the team decided to first focus on the customer dataset for the initial project development, since it was relatively clean and contained sufficient variables which allowed the team to concentrate on building meaningful customer insights rather than spending excessive time on heavy data processing since transaction type was not main focus.

1.2 Data Cleaning and Preparation

  • Li Gen suggested that the team should review the data quality of the customer dataset, including checking missing values, validating variable ranges, and standardising categorical variables and duplicate records.

  • Ricardo thought that the location information in the dataset is currently difficult to use directly for analysis and suggested converting the location variables into latitude and longitude coordinates so that they can be used for geographic visualisations.

  • Bingqian agreed that transforming the location data would allow the team to create maps and spatial visualisations in the future dashboard. Also noted that date-related variables should be converted into proper date formats so that they can be used for further analysis.

1.3 Project Plan and Analytical Storyline

  • Ricardo shared several possible analytical topics that the group could explore, including financial inclusion, customer lifetime value (CLV), churn analysis, customer satisfaction, cross-selling opportunities, and digital engagement behaviour.

  • Bingqian suggested that the project could be structured around customer analytics to create a clear and logical storyline, such as geographic adoption, user transaction behavior, customer segmentation and churn risk.

  • Li Gen proposed organising the analysis intothree analytical modules so that each member could focus on a different analytical component:
    Module 1: Customer Overview / Segmentation
    Understanding who the customers are and how they are distributed across demographic and product categories.
    Module 2: Customer Engagement and Experience
    Exploring how customers interact with the platform and analysing satisfaction and service issues.
    Module 3: Retention Risk and Customer Value
    Identifying high-value customers and customers with higher churn risk.

  • The team agreed that these modules form a progressive analytical workflow, starting from customer structure, moving to engagement behaviour, and ending with retention and business value insights.

1.4 Task Allocation

  • Ricardo volunteered to handle the location-related data cleaning, including converting location values into latitude and longitude coordinates and work on Module 3: Retention Risk and Customer Value, focusing on churn probability and customer lifetime value.

  • Li Gen will work on Module 2: Customer Engagement and Experience and assist with checking missing values and duplicate records, focusing on analysing engagement behaviour and satisfaction indicators.

  • Bingqian will focus on Module 1: Customer Overview / Segmentation, analysing customer demographics, acquisition channels, and product adoption patterns.

1.5 Next Steps

  • Li Gen will prepare the first draft of the project proposal, outlining the overall storyline, dataset description, and module structure.

  • Ricardo will continue reviewing the dataset and start working on data preparation tasks, particularly checking data quality and preparing variables for analysis.

  • Bingqian will begin exploring possible visualisation ideas for the Customer Overview & Segmentation module and document the meeting minutes.

  • The team agreed to review the first draft proposal together and refine the analytical storyline before the submission deadline.

2.2 Predictive Modelling and Dataset Performance Challenges

  • Ricardo raised a concern regarding the performance of predictive modelling in Module 3. When running the full customer dataset of 48,723 records with machine learning models such as Random Forest, the computation time was significantly long and caused the R session to become unresponsive in some cases.

  • The team discussed possible approaches to address this issue:

    • Reduce the dataset size by applying random sampling (e.g., 10,000 to 15,000 records) for model training and evaluation within the Shiny app, while using the full dataset for descriptive visualisations.
    • Switch to a lighter model such as logistic regression or decision tree for the Shiny prototype, and reserve more complex models for the report write-up only.
    • Pre-compute model outputs and cache the results as RDS files so that the Shiny app loads stored results rather than rerunning the model each time.

  • After discussion, the team reached a consensus that Ricardo will implement a combination of random sampling and result caching for the Shiny prototype. The full model results will be documented and discussed in the module write-up report rather than computed live in the app.

  • Bingqian noted a similar issue in Module 1, where the city-level behavioural heatmap using heatmaply took a noticeable amount of time to render when all cities were included. The team agreed to limit the heatmap to the top 30 cities by customer count to improve responsiveness.

2.3 GitHub Collaboration Issues and Resolution

  • The team encountered several issues during GitHub collaboration in the early development phase:
    - Merge conflicts: Li Gen and Bingqian both made changes to the _quarto.yml configuration file simultaneously, resulting in merge conflicts when pushing to the main branch. The conflict was resolved by manually reviewing the differences and merging both sets of changes.
    - Large file push failure: Bingqian encountered a push rejection because the .RDataTmp temporary file  was accidentally included in a commit, exceeding GitHub’s 100 MB file size limit. This was resolved by removing the file from the Git history using git filter-branch and adding .RDataTmp to the .gitignore file.
    - Branch management: The team had initially been working directly on the main branch, which caused issues when multiple members pushed changes at the same time. Li Gen suggested that each member work on their own feature branch and create pull requests to merge into main, to reduce the risk of conflicts going forward.

  • The team agreed to adopt the following GitHub workflow going forward:

    • Each member works on a dedicated branch.
    • Pull requests are reviewed by at least one other team member before merging into main.
    • The .gitignore file will be updated to exclude temporary R files (.RData, .RDataTmp, .Rhistory) and large data files.

  • Li Gen will update the .gitignore and push the updated file to the main branch, and will also update the _quarto.yml render list to exclude the module_retention.rmarkdown legacy file to prevent rendering errors.

2.4 Task Allocation and Next Steps

3.2 User Guide Format and Structure

  • Bingqian shared a reference example from a prior group project that structured the User Guide as a slide-based document, where each slide contains an annotated screenshot of the interface with numbered callouts and a description box explaining the functions of each UI element.

  • The team discussed two possible formats for the User Guide:
    - Option A — PowerPoint slide deck: Each slide covers one screen or tab. Screenshots are annotated with numbered labels and a side panel explains each element. This format is visually clear and easy for new users to follow.
    - Option B — Quarto HTML document: A scrollable web page with embedded screenshots, step-by-step instructions, and in-line annotations. This format integrates directly with the project website.

  • After discussion, the team agreed to produce the User Guide as a slide document with screenshot annotations following the same style as the slide-based reference. This would ensure the User Guide is accessible online and consistent with the rest of the website.
